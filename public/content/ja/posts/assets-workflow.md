元の大容量画像を GitHub リポジトリに入れたまま、サイトがまだ鮮明な WebP を出力するように保ちます。この投稿は `compress-assets` スクリプトとディレクトリ規約を記録しています。

||{[(ヒントの種類を区別するためのタイトルラベル付きの2番目の注釈。)]}画像資産|

## エンドツーエンドフロー

```
raw-assets/images/     ← 元の大容量画像をここに配置（プッシュしない）
        ↓
  node scripts/compress-assets.mjs
        ↓
public/assets/images/  ← 圧縮後の WebP（GitHub にプッシュ）
```

ディレクトリの準備ができたら、圧縮は次のコマンドだけで済みます：

```bash
npm run compress
```

`npm run compress` を実行すると自動的に：

1. `raw-assets/images/` で新規または変更されたビットマップファイルをスキャン。
2. 形式（PNG、JPEG、GIF など）ごとに品質プリセットを選択。
3. `public/assets/images/` に損失または無損失 WebP を書き込み。
4. 元の画像を Git から除外し、リポジトリ容量を制御。

| 元の形式 | 出力形式 | 品質 | 説明 |
| ------ | ------ | ---- | ---- |
| PNG    | WebP   | 90%  | 無損失圧縮、UI 素材向け |
| JPEG   | WebP   | 82%  | 損失圧縮、写真向け |
| WebP   | WebP   | 85%  | 統一品質 |
| AVIF   | WebP   | 85%  | トランスコード |
| GIF    | WebP   | 85%  | アニメーション GIF は単フレームに変換（アニメーションは保持されない）|

<circle-question>
このパイプラインはアニメーション GIF を単フレーム WebP に変換します。アニメーションが重要な素材であれば別途保存するかスクリプトを拡張してください。
</circle-question>

## ワンスクリーンセットアップ

Sharp をインストールし、元の資産ディレクトリを作成：

```bash
npm install sharp
mkdir raw-assets/images
```

次に PNG/JPEG/WebP/AVIF/GIF ソースファイルを `raw-assets/images/` に入れてスクリプトを実行。

## なぜ常に Unsplash を使うのですか？

プロトタイプ段階では外部リンクは問題ありません。長年にわたってメンテナンスされるサイトでは、ローカル最適化リソースの方が安定しています：

- 外部リンクはポリシー変更、レート制限、またはオフラインになる可能性があります。
- セルフホスト WebP はアセットタイプ（UI / 写真）ごとに品質を調整可能。
- CI で元の資産ディレクトリの欠落を検出すると、効果的な保護策として機能。

## .gitignore で元のファイルを除外

うっかり `git add .` で数メガバイトのソース画像をプッシュしないように、元ディレクトリを無視：

```gitignore
raw-assets/
```

> 大きなファイルをバージョン履歴に含める必要がある場合は、バイナリを通常のコミットに堆积するのではなく、Git LFS またはオブジェクトストレージを使用してください。
